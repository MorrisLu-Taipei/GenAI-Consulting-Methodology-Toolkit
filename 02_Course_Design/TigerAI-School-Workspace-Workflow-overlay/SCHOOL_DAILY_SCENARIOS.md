## 1. 颱風停課通知 / Typhoon School Closure

| 欄位 / Field | 內容 |
|---|---|
| **觸發者 / Triggered by** | 教育局 06:00 公告停課，校長 07:00 看到新聞 |
| **時間壓力 / Time pressure** | 1 小時內全校通知（家長要趕在上學前知道） |
| **對應 use case** | 公文路由 + 家長通知（兩者並行） |
| **涉及人員** | 校長、教務主任、學務主任、IT、全體導師、1,500 個家庭 |

### 流程（before AI）/ Before AI

```
07:00 校長看新聞 → 通知教務主任
07:05 主任手寫公文初稿（10 分鐘）
07:15 校長改 + 親簽（5 分鐘）
07:20 主任 LINE 傳給教師群組（教師再 forward 給家長）
07:40 部分家長收到（資訊雜亂、版本不一致）
08:30 家長 LINE 群組炸：「到底停課還是不停課？」
```

### 流程（with AI + HITL）/ With AI + HITL

```
07:00 校長 LINE 教務主任「教育局公告停課」
07:02 主任打開 Apps Script 觸發 "停課通知" workflow
       → AI 出 3 版草稿（正式 / 友善 / 簡短）+ 多語（中 / 英 / 越 / 印）
07:05 校長手機看到 approval request，挑「友善版」+ 親按「核可」
07:06 系統自動：
       → 校網公告 ✓
       → 教師群組 ✓
       → 導師 LINE 群組 ✓（但等導師看過再轉發給家長）
07:15 50 位導師各自審 1 次「給家長的 LINE 版本」，按「送出」
07:30 95% 家庭已收到統一訊息
08:00 家長 LINE 群組安靜，少數疑問導師可立刻回應
```

### AI 做什麼 / What AI does

- 從 1 句教育局公告 → 5 版草稿（風格 × 4 語言）
- 自動套校網模板 + LINE 模板
- 自動產生「導師對家長的個別化開頭」（XX 同學的家長您好）

### HITL Gate 在哪 / HITL Gate

- **校長手機 approval**：必須親按「核可」才會發到任何通道
- **導師 LINE 給家長前**：每位導師必須在 LINE Bot 介面按「確認送出」
- **絕對紅線**：AI 不可繞過任一步直接發

### 時間對比 / Time comparison

| 項目 | Before | After |
|---|---|---|
| 從新聞到全校 95% 知道 | 90 min | 30 min |
| 主任 + 校長投入 | 30 min | 5 min |
| 導師投入 | 5 min × 50 人 = 250 min | 1 min × 50 = 50 min |
| 家長混淆事件 | 高（版本不一） | 低（版本統一） |

### 典型失敗 / Common failure

> M4 真實事件：曾發生 AI 草擬版本含錯誤日期，導師按送出沒看到。改進：UI 強制顯示「AI 草稿」+「最終版」對比 + 5 個必勾項目。詳見 [`../SAMPLE_CLIENT_CASE_SCHOOL_K12.md`](../../04_Scenarios/SAMPLE_CLIENT_CASE_SCHOOL_K12.md) §5 M4 紀錄。

## 2. 戶外教學申請 / Off-Campus Event Application

| 欄位 | 內容 |
|---|---|
| **觸發者** | 七年級導師 想帶班去博物館 |
| **時間壓力** | 出發日前 14 天送件 |
| **對應 use case** | 活動申請 |
| **涉及人員** | 申請導師、學年主任、教務、學務、總務、會計、家長代表 |

### 流程

```
1. 導師打開 Google Form「戶外教學申請」
   AI 自動帶入：教師名、班級、學期已有活動次數
2. 導師填：目的地、日期、預算、安全規劃、學習目標
   AI 自動檢查：
     - 預算 / 班費餘額是否充足？
     - 是否與既有重要考試衝突？
     - 是否需特殊保險？
3. AI 出第一版簽核版本，路由至 學年主任
4. 學年主任在 GAS dashboard 看，點「同意」
5. AI 路由至 教務主任 → 學務主任 → 總務主任（依預算）
6. 預算 > NT$ 5,000 → 校長必簽
7. 全部簽完 → AI 自動：
   - 寄 confirmation email 給導師
   - 加進學校行事曆
   - 通知家長代表
   - 啟動家長同意書 workflow
```

### AI 做什麼

- 預算對照（接 Sheets 班費表）
- 衝突偵測（接學校行事曆）
- 多階段路由（依預算金額決定簽核者鏈）
- 同意書範本自動生成

### HITL Gate

- 每一站簽核者必須親點「同意 / 退回 / 修改」
- 預算 > NT$ 5,000 → 校長必簽
- 涉及校外風險（爬山 / 戲水）→ 學務主任 + 校長雙簽
- AI 不可代簽，不可跳站

### 時間對比

| 項目 | Before | After |
|---|---|---|
| 申請送到核可 | 5-7 天 | 1-2 天 |
| 紙本傳遞 | 5 站 × 1 天 = 5 天 | 0 |
| 卡在某主任桌上 | 常發生 | 系統自動提醒 24 hr |

### 典型失敗

- 「AI 沒抓到本週段考衝突」→ 改進：行事曆 integration 加上「重要事件」標籤過濾
- 「家長簽同意書沒回收」→ 改進：加 reminder workflow，3 天 / 1 週 / 1 天前各提醒一次

## 3. 教師請假 + 代課安排 / Teacher Leave + Substitute

| 欄位 | 內容 |
|---|---|
| **觸發者** | 教師早上 07:00 突然發燒 |
| **時間壓力** | 1 小時內要找代課 |
| **對應 use case** | 公文路由 |
| **涉及人員** | 請假教師、教務主任、學年主任、代課教師、影響的學生 |

### 流程

```
07:00 教師 LINE 學年主任「我今天發燒不能來」
07:01 主任打開「教師請假 + 代課」Apps Script workflow
07:02 AI 自動：
   - 抓該師今日課表（接 Workspace 課表）
   - 查同學科代課教師清單 + 今日 availability
   - 排出代課可能名單（含優先級）
07:05 主任挑 3 個候選人，LINE 詢問是否願意代
07:10 1 位答應
07:11 AI 自動：
   - 寫請假單草稿（病假類別 + 病情敘述模板）
   - 寄代課通知 + 該堂教案連結（接該師 Drive）
   - 通知班導 + 影響班級的家長群組
07:15 主任 + 校長線上簽請假單
07:30 代課教師收到完整資訊
```

### AI 做什麼

- 課表查詢（接 Workspace Calendar）
- 代課候選人排序（依學科 + 經驗 + 上週代課次數平均）
- 教案連結自動帶（接該師 Drive 共享資料夾）
- 多通道並行通知

### HITL Gate

- 代課老師必須親自接受（LINE 上按按鈕，不是 AI 代答應）
- 請假單必須請假教師、校長簽
- 病情細節（敏感）→ AI 不自動寫詳細，只寫「病假」
- 對家長通知**只說「今日由 OO 老師代課」，不揭露原因**

### 時間對比

| 項目 | Before | After |
|---|---|---|
| 找到代課 | 30-60 min（手機到處問） | 10 min |
| 代課教師拿到教案 | 通常沒拿到，臨場應變 | 課前 30 min 拿到 |
| 家長知道 | 不一定 | 同步知道 |

### 典型失敗

- 「AI 排的代課候選人都剛好不在」→ 改進：每月一次教師「願意代課時段」更新
- 「教案連結權限沒設好，代課老師打不開」→ 改進：onboarding 時設定 Drive 預設分享規則

## 4. 跨班物資請購 / Cross-Class Procurement

| 欄位 | 內容 |
|---|---|
| **觸發者** | 三班導師同週都要訂教具 |
| **時間壓力** | 不急，但累積拖久變不滿 |
| **對應 use case** | 物資請購 |
| **涉及人員** | 申請教師、採購組、會計、總務主任 |

### 流程

```
W1 D1：導師 A 提請購：A4 紙 5 包
W1 D2：導師 B 提請購：A4 紙 3 包
W1 D3：導師 C 提請購：A4 紙 7 包
        AI 偵測：3 筆都是「A4 紙」，自動建議合併採購
W1 D4：採購組同意合併 → AI 出整合請購單（15 包）
        → AI 對照常用品庫存：已有 5 包，建議只購 10 包
        → 路由給總務主任
W1 D5：總務主任簽 → 進採購流程
W2 D2：到貨 → AI 通知 3 位導師「你訂的 A4 紙到了」
```

### AI 做什麼

- 跨時間 / 跨教師整合相同物品
- 庫存對照（接 Sheets 庫存表）
- 預算對照
- 廠商比價（如有歷史採購資料）

### HITL Gate

- 採購組必須親按「同意合併」（不可自動合併）
- 採購金額 > NT$ 10,000 → 總務主任必簽
- 廠商選擇必須人工確認（AI 不代選）

### 時間對比

| 項目 | Before | After |
|---|---|---|
| 請購到到貨 | 平均 2-3 週 | 1-1.5 週 |
| 整合採購省下成本 | 0 | ~ 15-20% |
| 採購組額外時間 | 整合手工 | 0 |

### 典型失敗

- 「AI 把不同規格 A4 紙視為相同」→ 改進：欄位加上「規格」必填 + AI 比對規格再合併
- 「合併後其中 1 人臨時取消」→ 改進：合併前再次確認所有人都還要

## 5. 家長投訴信回覆 / Parent Complaint Response

| 欄位 | 內容 |
|---|---|
| **觸發者** | 家長 LINE 投訴「我小孩被某老師罵」 |
| **時間壓力** | 24 小時內首次回應，3 天內結案 |
| **對應 use case** | 家長通知（**高敏感**） |
| **涉及人員** | 學務主任、班導、被投訴老師、輔導室、家長 |

### 流程

```
T+0  家長 LINE 訊息進來
       AI 自動：
         - 標記 sentiment（負面 / 急 / 涉及人事）
         - 通知學務主任 + 輔導室主任
         - 不自動回覆
T+30min 學務主任打開案件，AI 整理：
         - 家長過往訊息歷史
         - 該學生的相關紀錄（去識別化處理）
         - 該老師的相關紀錄（去識別化處理）
         - 過往類似投訴的處理模式
T+1h   學務主任跟班導電話 + 跟被投訴老師談
T+2h   AI 草擬 3 版回覆（同理優先 / 中性說明 / 道歉版）
T+3h   學務主任挑一版 + 大幅手動修改 + 校長 review
T+4h   學務主任親自打電話給家長 + 同步 LINE 文字
T+24h  正式書面回覆（紙本 + LINE）
T+3d   後續追蹤
```

### AI 做什麼

- Sentiment + 緊急度分類
- 歷史紀錄整理（**去識別化處理**）
- 多版本草稿
- **不**直接回覆家長

### HITL Gate（紅線最嚴）

- ⚠️ **任何敏感投訴回覆必須學務主任 + 校長雙審**
- ⚠️ **AI 不接觸學生未去識別化的姓名 / 個資**
- ⚠️ **AI 不對家長直接發送任何訊息**
- ⚠️ 第一次回應**必須電話（不是純文字）**
- ⚠️ 涉及人事 / 法律 → 立刻 escalate 校長 + 法務

### 時間對比

| 項目 | Before | After |
|---|---|---|
| 學務主任查歷史紀錄 | 30-60 min | 5 min |
| 草擬回覆 | 30 min | 5 min（AI 出，主任改）|
| 整體案件 turnaround | 不變（人為決策時間是 floor） | 不變 |

> **這個情境 AI 不是為了省時間，是為了「整理資訊讓人能更好判斷」。** Time savings is NOT the point here — better-informed human decision IS.

### 典型失敗

- 「AI 把學生名字洩漏在草稿裡」→ 改進：強制去識別化 + 草稿不可含未脫敏資料
- 「主任改太少就送出」→ 改進：UI 顯示「diff」+ 「你只改了 5%，確定要送嗎」warning

## 6. 學生違規通知家長 / Student Disciplinary Notification

| 欄位 | 內容 |
|---|---|
| **觸發者** | 學生 X 在上課時打架 |
| **時間壓力** | 當天通知家長 |
| **對應 use case** | 家長通知（**含個資 + 兒少法**） |
| **涉及人員** | 班導、學務主任、輔導室、家長 |

### 流程

```
事件當下：班導記錄事件（紙本 or LINE）
休息時間：班導打開「學生事件記錄」workflow
         填入：學生代號（不寫姓名）+ 事件類別 + 簡述
         AI 自動：
           - 對應學校《校規》條款
           - 對應過往類似事件的處理（去識別化）
           - 建議處理路徑（輕度 / 中度 / 嚴重）
         如有「自傷 / 受虐」關鍵字 → **AI 立刻 alert 輔導室，不繼續處理**
午休：班導決定處理路徑 + 跟學務主任面談（不是 AI）
下午：班導用 AI 草擬給家長通知（事實 + 同理 + 後續安排）
傍晚：班導 + 學務主任雙審 → 班導打電話給家長（**不是純文字**）
晚上：紙本通知書 + LINE 文字（已 talked 過電話後）
```

### AI 做什麼

- 校規條款對應
- 過往類似事件處理檢索（**全去識別化**）
- 處理建議
- 通知草稿

### HITL Gate（兒少法 + 個資法雙重紅線）

- ⚠️ AI 不接觸學生真實姓名（系統用學生代號）
- ⚠️ 自傷 / 受虐 / 性別議題 → **AI 不處理，直接 alert 輔導室專業人員**
- ⚠️ 通知家長**必須電話為主，文字為輔**
- ⚠️ 「停學 / 退學 / 通報社政」級別 → **校長 + 法務 + 社工** 多方參與，AI 完全退出

### 時間對比

| 項目 | Before | After |
|---|---|---|
| 找校規條款 | 15-20 min（翻紙本）| 1 min |
| 寫通知 | 30 min | 10 min |
| 重大事件處理品質 | 視主任經驗 | 經 AI 整理過往案例後判斷較完整 |

### 典型失敗

- 「AI 漏 alert 含自傷關鍵字的事件」→ 改進：關鍵字清單持續更新 + 輔導室 review
- 「家長收到 LINE 先看到，沒先電話」→ 改進：workflow 強制電話 step 先完成才能發 LINE

## 7. 親師會多語通知 / Multilingual Parent-Teacher Conference

| 欄位 | 內容 |
|---|---|
| **觸發者** | 學期中親師會排定 |
| **時間壓力** | 親師會前 2 週發出 |
| **對應 use case** | 家長通知（多語） |
| **涉及人員** | 教務主任、行政組長、全體導師、家長（含外籍家庭 ~10%） |

### 流程

```
D-14  教務主任發 LINE 給行政組長「親師會通知請發」
D-14  AI 自動：
        - 草擬通知模板（時間 / 地點 / 議程 / 報名連結）
        - 出 4 語版本：中 / 英 / 越 / 印
        - 帶入每班導師簽名 + 班級時段
D-13  教務主任 + 各語言能力者審多語版本
        英語老師審 EN；新住民協助老師審越 / 印
D-12  各班導師收到「您班的通知預覽」，導師審 + 改 + 按送出
D-11  AI 自動分發：
        - 中文家庭收中文版
        - 外籍家庭收對應母語版
        - LINE / Email 雙通道
D-10  AI 追蹤已讀 / 未讀
D-7   未報名家長收到提醒
D-3   再次提醒未報名家長
D-0   親師會
```

### AI 做什麼

- 多語翻譯 + 風格統一
- 家長語言偏好識別（接 SIS 家長語言欄位）
- 已讀追蹤
- 提醒節奏自動化

### HITL Gate

- **各語版本必須該語言能力者親審**（AI 翻譯誤差不可不審）
- **每位導師必須個別審該班版本**
- 對未報名家長提醒**頻率合理**（避免騷擾）

### 時間對比

| 項目 | Before | After |
|---|---|---|
| 通知撰寫 | 3 hr | 30 min |
| 翻譯 | 找翻譯公司 2-3 天 | 即時 + 1 hr 人工審 |
| 報名追蹤 | 行政組長一個個打電話 | 系統自動 |
| 報名率 | 60-70% | 80-90% |

### 典型失敗

- 「越南文翻譯有錯，新住民協助老師發現晚了」→ 改進：審查 SLA 明確（D-12 必審完）
- 「外籍家長收到中文版」→ 改進：SIS 語言偏好欄位定期 audit

## 8. 招生諮詢 FAQ / Admissions Inquiry FAQ

| 欄位 | 內容 |
|---|---|
| **觸發者** | 家長透過校網 / LINE 問入學問題 |
| **時間壓力** | 24 小時內回應 |
| **對應 use case** | 候補 use case（M9 後考慮） |
| **涉及人員** | 招生組、教務主任 |

### 流程

```
家長提問（透過校網表單 or LINE）
  ↓
AI 自動分類：
  - 學費 / 學區 / 課程 / 設施 / 申請流程
  ↓
判斷信心度：
  - confidence ≥ 0.85 + 在 FAQ 庫內 → AI 直接回（但標註「AI 助理回覆」）
  - 0.5-0.85 → AI 出草稿給招生組
  - < 0.5 → 不出草稿，直接 escalate 招生組
  ↓
招生組或 AI 回家長
  ↓
家長回追問 → AI 標 sentiment + escalation 判斷
```

### AI 做什麼

- 自動分類 + 信心度評估
- 引用 FAQ 庫回答
- 多語回覆（外籍家庭）
- 統計常見問題（給招生組看趨勢）

### HITL Gate

- ⚠️ AI 自動回答**必須標註**「本訊息經 AI 助理回覆，由 OO 老師審查」
- ⚠️ 涉及個別孩子情況（如「我小孩有過動症能來嗎」）→ 立刻 escalate
- ⚠️ 涉及學費 / 入學考 / 法律問題 → 必須真人回
- ⚠️ AI 回答**不可作為承諾**，UI 顯示「正式回覆以招生組為準」

### 時間對比

| 項目 | Before | After |
|---|---|---|
| 招生組每天處理 | 30-50 個問題 | 5-10 個（80% AI 處理） |
| 平均回應時間 | 12-24 hr | 5 min（AI）/ 8 hr（人） |
| 家長滿意度 | 7.5 | 8.7 |

### 典型失敗

- 「AI 答錯學費結果家長帶錢來」→ 改進：學費類問題一律不 AI 回，強制 escalate
- 「家長以為 AI 是真人」→ 改進：所有 AI 回覆強制 prefix「【AI 助理】」

## 9. 8 情境總對照表 / 8 Scenarios Comparison Table

| # | 情境 | use case | HITL 嚴格度 | 時間節省 | 紅線數 |
|---|---|---|---|---|---|
| 1 | 停課通知 | 公文+家長通知 | ⚠️ 中（校長必簽） | 90→30 min | 3 |
| 2 | 戶外教學申請 | 活動申請 | ⚠️ 中（多階段簽） | 5-7d→1-2d | 2 |
| 3 | 教師請假 + 代課 | 公文 | ⚠️ 中（代課必接受） | 60→10 min | 3 |
| 4 | 跨班物資請購 | 物資請購 | 🟢 低（採購組） | 2-3w→1-1.5w | 1 |
| 5 | 家長投訴回覆 | 家長通知 | 🔴 **極高**（雙審 + 電話） | **不省時間** | 5 |
| 6 | 學生違規通知 | 家長通知 | 🔴 **極高**（兒少法） | 略省 | 5 |
| 7 | 親師會多語通知 | 家長通知 | ⚠️ 中（多語審） | 大幅省 | 3 |
| 8 | 招生 FAQ | 候補 use case | 🟡 中低（標註 AI） | 12-24h→5 min | 4 |

**結論：**
- **L3 行政自動化 ≠ 全自動**。**5 個情境**中 HITL 是「主體不是配角」。
- **時間節省最大的是「重複性高 + 敏感度低」**（場景 1, 7）；「敏感度高」場景的 AI **價值不在省時間，在於資訊整理讓人能更好判斷**（場景 5, 6）。
- **每個情境的 HITL 紅線數 ≥ 1**。沒有紅線的 use case 在學校環境**不存在**。

**Conclusion:**
- **L3 admin automation ≠ full automation**. In **5 of 8** scenarios, HITL is the main mechanism, not a footnote.
- **Biggest time saving comes from "high-repetition + low-sensitivity"** (scenarios 1, 7); **high-sensitivity scenarios get AI value not from time savings but from information consolidation enabling better human judgment** (scenarios 5, 6).
- **Every scenario has ≥ 1 HITL red line**. In school context, "no-red-line use case" **does not exist**.

## 10. 怎麼用本檔 / How to use this file

| 角色 / Role | 怎麼用 / How to use |
|---|---|
| **顧問接案** | 跟校長簡報時拿 1-2 個情境（如停課 + 家長投訴），展示「AI 不是要取代人，是要讓人更專業」 |
