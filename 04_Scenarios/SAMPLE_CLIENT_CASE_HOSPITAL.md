# 範例客戶完整案例：醫院 / 醫療機構

更新日期：2026-05-13

## 1. 案例定位

本案例示範醫院如何使用「企業 AI 轉型成熟度診斷與導入方案」，在高敏感資料與高度法規要求下，從 AI 使用盤點、課程能力建置，到顧問診斷報告形成可落地 Roadmap。

案例產業：醫療機構 / 區域醫院  
公司規模：約 1,500 人  
主要部門：醫務行政、護理、門診、客服中心、病歷室、資訊室、品質管理、財務、人資  
主要系統：院內 Email、Excel / Sheets、SharePoint / Notion、HIS、EMR、LIS、PACS、掛號系統、客服系統、內部 API  
建議部署：全地端或嚴格 Hybrid  
核心原則：病患資料、病歷、檢驗、影像與個資不得直接上公有雲；所有臨床相關輸出必須人工審核

## 1.1 案例交付邏輯

本案例不是要證明醫院可以立刻做醫療診斷 AI，而是示範如何在高敏感資料環境下，用顧問方法安全推進 AI。

完整交付流程：

1. 現況資料蒐集：盤點院內系統、資料分級、AI 使用現況與法規限制。
2. 問卷診斷：完成 10 題快速診斷與 25 題課前診斷。
3. 風險評估：判斷可自動、需審核、禁止的 AI 場景。
4. 訪談評估：訪談醫務行政、客服中心、護理、資訊室、品質管理。
5. As-Is 流程盤點：繪製病患服務客服流程與院內 SOP 查詢流程。
6. L1-L5 能力評估：逐級定義 input、process、output、evidence。
7. 課程與工作坊：先做 L1 治理與 L2 Skill，再做 L3 低風險 Workflow。
8. 顧問診斷報告：產出部署建議、治理設計、Roadmap、ROI 與後續 SOW。

每一階段都必須留下可驗證 deliverables，尤其是醫療場景必須保留 Log、人工審核與責任歸屬。

## 2. 客戶故事

這家醫院已經有部分行政與醫護同仁開始嘗試使用 AI。行政人員會用 AI 改寫公告、整理會議紀錄；客服人員會用 AI 草擬回覆；資訊室也聽過 n8n、自動化與 Agent。

但醫院管理層非常謹慎：

- 病患資料不能外流。
- 醫療建議不能由 AI 自動決策。
- 醫護人員工作繁忙，沒有時間學複雜工具。
- 內部 SOP、公告、規章、病患服務流程很多，但查找效率低。
- 客服中心常被重複問題佔用，例如掛號、看診流程、報到、檢查前注意事項。
- 資訊室擔心 AI 導入後變成更多維運負擔。

因此本案的策略不是一開始做醫療診斷 AI，而是先從低風險、高重複、高行政負擔的場景開始：

1. 院內 AI 使用入口與規範。
2. 醫院行政與客服 Skill 化。
3. n8n 串接客服、知識庫、表單、內部 API。
4. Hermes Agent 作為行政與客服輔助，不直接做臨床決策。
5. 後續再評估 ClawTeam 用於品質管理、病患服務改善與行政專案協作。

## 4. 公司屬性與部署模式判斷

### 4.1 公司屬性

| 項目 | 判斷 |
| --- | --- |
| 產業 | 醫療 / 醫院 |
| 資料敏感度 | 極高，包含病患個資、病歷、檢驗、影像、費用資料 |
| 法規與稽核 | 高，需符合個資、醫療資料治理、內部稽核要求 |
| 系統封閉度 | 高，HIS、EMR、LIS、PACS 多為內網核心系統 |
| IT 能力 | 中高，有資訊室，但維運負擔大 |
| 雲端政策 | 核心醫療資料不得上雲，一般行政資料需審核 |
| 導入風險 | 個資外洩、錯誤醫療建議、權限誤用、責任歸屬不清 |

### 4.2 建議部署模式

建議採全地端或嚴格 Hybrid。

部署原則：

- 病患個資、病歷、檢驗、影像與 HIS / EMR 資料不得直接送公有雲。
- L1 OpenWebUI 建議使用院內受控環境。
- L2 Skill 可先處理行政 SOP、客服 FAQ、院內規章、表單流程。
- L3 n8n 串接時需以查詢權限最小化為原則。
- L4 Hermes Agent 初期只做行政輔助、客服摘要、內部查詢，不做醫療診斷或治療建議。
- 所有對病患輸出的內容都需人工確認。
- 所有流程需保留 Log、輸入、輸出、操作者與時間戳。

## 5. 建議課程比例

| 模組 | 比例 | 原因 |
| --- | ---: | --- |
| L1 OpenWebUI | 30% | 建立受控入口、AI 使用規範、資料分級與安全意識 |
| L2 Skill AI | 35% | 將行政 SOP、客服 FAQ、病患服務流程轉為可複用 Skill |
| L3 n8n Workflow AI | 25% | 從低風險行政與客服流程開始做 PoC |
| L4 Hermes Agent | 10% | 設計行政 / 客服輔助 Agent，但保留人工審核 |
| L5 ClawTeam | 0% | 暫不進入多 Agent 團隊，待治理成熟後再規劃 |

課程產品包建議：一日工作坊 + 顧問診斷專案  
若資訊室與管理層參與度高，可升級為二日導入營

## 6. 課程中應產出的工作資產

### 6.1 L1 產出

- 醫院 AI 使用規範草案。
- 資料分級原則：公開、內部、敏感、醫療核心資料。
- OpenWebUI 院內使用流程。
- 行政與客服常用 Prompt。

### 6.2 L2 產出

優先建立 5 個低風險 Skill：

1. 院內公告改寫 Skill。
2. 會議紀錄整理 Skill。
3. 病患服務 FAQ Skill。
4. 掛號與報到流程說明 Skill。
5. 檢查前注意事項摘要 Skill。

注意：檢查前注意事項 Skill 必須引用院方核准內容，不可由 AI 自行產生醫療判斷。

### 6.3 L3 產出

優先完成 1 個 PoC：

> 病患服務 FAQ 與客服案件整理 Workflow

流程：

1. 客服系統或 Email 收到病患服務問題。
2. AI 判斷問題類型，例如掛號、報到、批價、檢查注意事項、文件申請。
3. 從院內核准 FAQ / SOP 知識庫查詢答案。
4. 若涉及個人病情、病歷、檢驗或醫療建議，標記為人工處理。
5. 產生客服回覆草稿。
6. 人工確認後回覆。
7. 將案件分類與處理結果寫入 Sheets 或客服系統。

### 6.4 L4 產出

Hermes Agent 任務卡：

名稱：病患服務客服輔助 Agent  
目標：協助客服人員分類問題、查詢核准 FAQ、產生回覆草稿與追蹤案件。  
可用工具：病患服務 FAQ Skill、掛號流程 Skill、檢查注意事項摘要 Skill、n8n 客服 Workflow、知識庫查詢。  
限制：不得提供診斷、治療、用藥建議；不得查詢或輸出病患個資；涉及病情一律轉人工。  
輸出：問題分類、FAQ 來源、回覆草稿、人工處理標記、案件摘要。

## 7. 八階段顧問診斷分析

### Stage 1：As-Is Snapshot 現況掌握

觀察：

- 行政與客服有零星 AI 使用，但缺正式規範。
- 院內 SOP 與 FAQ 很多，但查找成本高。
- 病患服務問題重複性高。
- 資訊室對資料外流與維運風險高度敏感。

輸出：

- AI 使用盤點。
- 客服案件 As-Is Process Map。
- 資料分級與系統盤點。

### Stage 2：Reference Model Alignment 標準模型引入

判定：

- 整體 L1 尚未穩定。
- 行政、客服具備 L2 Skill 化潛力。
- L3 需從低風險流程開始，不直接碰核心醫療資料。

輸出：

- L1-L5 成熟度評估表。
- 醫院可導入 / 不可導入 AI 場景分類。

### Stage 3：Best Practice Integration 最佳實務對標

醫院優先 AI 場景：

- 院內公告與行政文件改寫。
- 會議紀錄與交辦事項整理。
- 病患服務 FAQ。
- 掛號、報到、批價流程查詢。
- 檢查前注意事項摘要。
- 內部 SOP 查詢。
- 品質管理事件摘要。

暫不建議第一階段處理：

- 醫療診斷。
- 治療建議。
- 用藥建議。
- 未經核准的病歷自動摘要對外輸出。
- Agent 直接修改 HIS / EMR。

### Stage 4：Gap Analysis 差距分析

| 類型 | 發現 | 影響 |
| --- | --- | --- |
| Missing | 缺 AI 使用規範 | 同仁不知道哪些資料可用 AI |
| Missing | 缺資料分級與 AI 場景分級 | 導入容易卡在資安疑慮 |
| Missing | SOP / FAQ 未 Skill 化 | 查找效率低，客服回答不一致 |
| Broken | 病患重複問題人工處理 | 客服負擔高 |
| Redundant | 行政公告、會議紀錄、FAQ 回覆重複撰寫 | 浪費行政時間 |
| Missing | 缺 Log 與稽核機制 | 不符合醫療場景治理要求 |

### Stage 5：Problem Definition 核心問題定義

Executive Problem Statement：

> 醫院目前 AI 使用仍停留在零星個人嘗試，雖然行政與客服場景具備高重複性與導入價值，但因缺少資料分級、AI 使用規範、可稽核 Workflow 與人工審核機制，使 AI 難以安全進入院內流程。

### Stage 6：Benchmarking and Phased Goals 階段目標

90 天目標：

- 建立院內 AI 使用規範與資料分級。
- 建立 OpenWebUI 或受控 AI 入口。
- 完成 5 個行政 / 客服 Skill。
- 完成 1 個病患服務 FAQ Workflow PoC。
- 建立 Hermes Agent 任務卡與人工審核流程。
- 建立 AI 使用 Log 與初步 KPI。

### Stage 7：To-Be Design 未來藍圖

To-Be 架構：

- L1：院內 OpenWebUI 受控入口。
- L2：行政與客服 Skill Library。
- L3：n8n 串接客服系統、FAQ 知識庫、Email、Sheets、內部 API。
- L4：Hermes Agent 作為病患服務客服輔助 Agent。
- L5：後續可規劃 ClawTeam，例如「品質改善 Agent Team」或「行政專案 Agent Team」。

### Stage 8：Implementation and Change 導入與變革治理

治理建議：

- AI 場景分級：可自動、需審核、禁止。
- 資料分級：公開、內部、敏感、醫療核心。
- 對外回覆一律人工確認。
- Agent 不得提供醫療診斷、治療或用藥建議。
- HIS / EMR / LIS / PACS 初期不直接開放 Agent 查詢個資。
- 所有 AI Workflow 保留 Log。
- 建立資訊室、法遵 / 稽核、業務 Owner 三方審核。

## 8. 三階段 Roadmap

### Phase 1：0-30 天，AI 治理與 Skill 化

任務：

- 建立 AI 使用規範。
- 建立資料分級與場景分級。
- 建立受控 AI 入口。
- 產出 5 個行政 / 客服 Skill。

驗收：

- 完成 20-30 位種子人員訓練。
- 完成 5 個低風險 Skill。
- 完成 AI 可用 / 禁用資料清單。

### Phase 2：31-60 天，客服 FAQ Workflow PoC

任務：

- 建立病患服務 FAQ 知識庫。
- 用 n8n 串接客服系統、Email、FAQ、Sheets。
- 建立人工審核節點。
- 建立 Log 與錯誤通知。

驗收：

- AI 可分類常見問題。
- AI 可引用核准 FAQ 產生回覆草稿。
- 涉及病情或個資自動轉人工。
- 產出客服案件統計。

### Phase 3：61-90 天，Hermes Agent 與治理擴大

任務：

- 建立病患服務客服輔助 Hermes Agent。
- Agent 可呼叫 FAQ Skill 與客服 Workflow。
- 建立 KPI Dashboard。
- 評估第二波 PoC：院內 SOP 查詢或品質事件摘要。

驗收：

- Hermes Agent 可產生問題分類、FAQ 來源與回覆草稿。
- 所有回覆保留人工確認。
- 管理層可看到客服處理效率與 AI 採用率。

## 9. ROI 指標

| --- | --- | --- |

| Skill 產出 | 無正式 Skill | 5 個低風險 Skill |
| Workflow PoC | 無 | 1 個客服 FAQ PoC |
| AI 使用治理 | 無正式紀錄 | 建立使用規範與 Log |

## 10. 最終顧問報告摘要

本案例的建議結論：

- 目前成熟度：整體 L1 尚未穩定。
- 目標成熟度：90 天內建立 L1 治理基礎，完成 L2 Skill 化與 1 個 L3 PoC。
- 建議部署：全地端或嚴格 Hybrid。
- 優先場景：病患服務 FAQ 與客服案件整理。
- 建議課程：一日工作坊 + 顧問診斷專案。
- 後續導入：病患服務客服輔助 Hermes Agent，第二波擴展到院內 SOP 查詢與品質事件摘要。

## 13. Stage Gate 驗收設計

### Gate 1：L1 受控入口與資料分級

必備證據：

- 院內 AI 使用規範完成。
- 資料分級清單完成。
- OpenWebUI 或受控 AI 入口可用。
- 禁止輸入病患個資與病歷資料的規範已公告。

### Gate 2：L2 低風險 Skill Library

必備證據：

- 至少 5 個低風險 Skill。
- 每個 Skill 均有核准來源文件。
- 每個 Skill 有 Owner 與版本紀錄。
- Skill 不自行產生醫療診斷、治療或用藥建議。

### Gate 3：L3 客服 FAQ Workflow

必備證據：

- n8n Workflow 可執行。
- 可查詢核准 FAQ / SOP。
- 涉及個資、病情、檢驗、治療、用藥的問題會自動轉人工。
- 有 Log、人工審核與錯誤通知。

### Gate 4：L4 Hermes Agent

必備證據：

- Hermes Agent 任務卡完成。
- Agent 可呼叫至少 1 個 Skill 與 1 個 n8n Workflow。
- Agent 權限明確，不可查病歷，不可給醫療建議。
- 所有對外回覆人工確認。

### Gate 5：L5 ClawTeam 準備度

必備證據：

- L1-L4 已有穩定治理與 Log。
- 已定義可用於多 Agent 的非臨床或品質改善任務。
- 已設計法遵 / 稽核 / 業務三方人工 Gate。
